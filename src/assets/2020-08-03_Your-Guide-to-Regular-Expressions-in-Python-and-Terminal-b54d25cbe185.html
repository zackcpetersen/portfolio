Your Guide to Regular Expressions in Python and Terminal

Step up your regex know-how

Regular expressions. If you know how to use them, they are awesome and powerful. If not, they look like an alien language that needs to be decoded. Either way, this guide will cover the basics of regexes and how they are used to make the life of a programmer easier. We will then go over the basic & extended set and how they can be used in Python and terminal. By the end, you’ll have enough regex know-how to step up your everyday coding.

The Basics

Regular expressions are a way to search for patterns in data sets. It is not a programming language on its own, but all major languages have a regex engine built-in for you to utilize.

What is a regular expression anyway?

“A regular expression is a sequence of characters that define a search pattern.” — Wikipedia

The “sequence of characters” are regex syntax, and if you’ve ever seen it you know how foreign it is to read, let alone understand, especially at first. For example:

^\(?[0-9]{3}\)?[\-\s]?[0-9]{3}[\-\s]?[0-9]{4}$

At first glance, this can be pretty overwhelming. Most of us don’t want to even try and figure it out. But, it’s just an expression to match a phone number. We’ll break this down in-depth later in the article, so don’t worry if you don’t understand it right now.

When to, and when not to, use regexes

Regexes are useful to programmatically:

  • Find patterns and matches in data sets: Whether it’s a .csv or .txt, they’ve got you covered
  • Validate forms and inputs: Do this on the frontend before you submit a form to make sure there are no issues behind the scenes once it’s been sent.
  • Scrape information from the web: Looking to scrape every link (<a></a>) off of a webpage? There’s an expression for that.

What are the limitations of regular expressions?

  • Regular expressions are not a programming language. They are tools and all major languages utilize them.
  • Difficult to read. Especially if you’re not familiar with them, it’s pretty intimidating and most people, myself included, tend to stay away from them for as long as possible.
  • Can be difficult to write correctly, even if you know what you’re doing. Sometimes getting it to work exactly how you want takes more than a few tries.

Different Types of Regular Expressions

At this point, we have a basic understanding of what regular expressions are and how they can be used. However, being able to write them on your own is a valuable tool to have. Let’s dive in.

Basic & extended set

Regexes have a basic set and an extended set. For this guide, we will cover the basic set and only parts of the extended set, to see an exhaustive list of what’s available, check out the regular expression operation docs for Python.

Basic Set Symbols

. — period (wildcard): replaces any character

^ — caret: matches the start of a string

$ — dollar sign: matches the end of a string

* — asterisk: matches the preceding expression zero or more times

\ — forward slash: represents special characters and escapes characters

() — parenthesis: groups expressions together

? — question mark: matches zero or one of the preceding expression

\s — escaped ‘s’: represents whitespace

[wxyz] — square brackets, literal: represents one of the characters inside (either w, x, y, or z)

[w-z]— square bracket, range: represents one of the characters inside the range (either w, x, y, or z ), can also be used with numbers [10–100].

[^wxyz] — square brackets, negated: represents any character not in the set (anything except w, x, y, or z). Note the caret inside the square brackets, has a different meaning than caret from above.

\w — word: matches any ASCII letter, digit, or underscore

Extended Set Symbols

+ — plus sign: matches the preceding expression one or more times

| — pipe: used as an “or” comparison operator. Will match either expression1 OR expression2: expression1|expression2

{n}— will match “n” occurrences of the preceding expression, similar to *, +, and ?, except if you define the max number of occurrences with n.

The main difference to keep in mind when using the basic set vs. the extended set will be in the terminal. grep, for example, will need to be followed by -E when using symbols from the extended set.

grep -E ‘your-extended-regex’

Breaking Down the Expression

Let’s revisit that expression from the beginning:

^\(?[0-9]{3}\)?[\-\s]?[0-9]{3}[\-\s]?[0-9]{4}$

To make this easier to read, let’s break it into smaller chunks and reverse-engineer it to show how it actually matches a phone number. Use the symbols labeled above as a guide. You can also check out RegExr and follow along to see matches as they are created.

^ — the caret: This is to signify the start of our string. Easy peasy.

\(? — optional open parenthesis: We are first escaping \ it so it will read as a literal character ( . Then we’ll add a ? to look for it either zero or one time, making it optional.

[0–9]{3} — looks for any three consecutive numbers, aka an area code (305, 801, 702).

\)? — optional closing parenthesis

At this point, the expression will match either 123 or (123).

[\-\s]? — optional whitespace or dash: \s is the expression for space or whitespace, inside the brackets with a \- (escaped dash), followed by a ?, so it will look for it zero or one time.

[0-9]{3} — looks for any three consecutive numbers

[\-\s]? — optional dash or whitespace.

[0-9]{4} — final expression before the ending tag, looks for any four consecutive numbers.

$ — end of the string.

In this case, both the ^ and the $ could be omitted and are not required for this specific expression. I left them in because they are useful to know and work either way for this scenario. If the phone number was in the middle of other text, you would want to leave them both out.

It’s not so bad when it’s broken down, right? This expression will match any of the following phone number formats:

  • 1234567891
  • 123 456 7891
  • 123 456–7891
  • 123–456–7891
  • (123) 456 7891
  • (123) 456–7891

We can simplify if we wish to ignore certain parts of the original expression:

[0-9]{10} — probably the simplest way to check, will not match if any dashes, spaces, or parenthesis are included. Only looks for consecutive ten-digit number.

[0-9]{3}[0-9]{3}[0-9]{4} — a different way to write a consecutive ten-digit number expression (this is essentially identical to the expression above), makes it easier to add in dashes or spaces.

[0-9]{3}[\-\s]?[0-9]{3}[\-\s]?[0-9]{4} — same as previous, with checks for dashes and spaces

Now, let’s look at one way to identify patterns and build our own regexes.

Regular Expression to Match an Email Address

At this point, we’ve got a basic understanding of some of the symbols and tools available for use in building these expressions. Let’s take a look at another common pattern we’re all familiar with, email addresses.

Before we start actually making the regex, let's look at a three-step process to help us identify the pattern we’d like to match with.

  1. What needs to be included or excluded?
  2. Find patterns in the inclusions/exclusions.
  3. Write those patterns in a RegEx.

Looking at these emails as an example:

  • regular-expression-master101@gmail.com
  • joe_smith@aol.com
  • business-man99@bigbusiness.net

Let’s apply the three-step process above to our email

  1. What needs to be included or excluded?
    - We’ll break up the emails into parts.
    <username>@<second-level-domain>.<top-level-domain>
    - The top-level domain could end in .com, .net, .org, etc. For the sake of this example, we are going to say that the max length will be four, (something like .info).
  2. Find patterns in the inclusions/exclusions
    - Username will need to be pretty open, but we won’t be allowing for special characters outside of underscores, dashes, and periods.
    - We’ll have a fixed @ symbol
    - The second-level domain will exclude special characters except for underscores, dashes, and periods.
    - The top-level domain must only be letters and have a max length of four and a min length of two.
  3. Write those patterns in a RegEx
    - Username, [\w\-\.]+, \w matches with any ASCII letter, digit, underscore. \- is a literal dash. \. is a period. Wrap those in square brackets to match any one of them, and add + after to include it one or more times.
    - Fixed @ symbol matches with the @ symbol and nothing else. This is required.
    - Second-level domain, ([\w\-]+\.)+ , similar to username expression, except this allows for multiple expressions to be strung together and is grouped with parenthesis () (example@myurl.co.uk)
    - Top-level domain, [a-zA-Z]{2,4}, allows for “A” through “Z,” both lower and uppercase, from two to four characters in length.

Put all of this together, with a caret at the beginning and a dollar sign at the end, and let's see what it looks like.

^[\w\-\.]+@([\w\-]+\.)+[a-zA-Z]{2,4}$

Beautiful. Throw that into RegExr and test it against each email individually, or remove the caret and dollar sign and test them all together. I also encourage you to add and remove parts of the regex to see if you can make it more or less flexible.

Learning regexes is great and all, but how do we actually apply it to our daily programming? Let’s look at how we can use regexes in terminal and Python.

Note:
Different regex engines handle expressions in their own ways. Sometimes, what works on regexr.com or in grep might not work in Python & vice-versa. You may have to experiment and try making your Regular Expressions in a few ways before it works as expected

Regular Expressions In Terminal

For this example, we’ll be using grep. Grep is a command-line utility used for searching plain-text data sets and it has its own regex engine built in! Let’s grab the phone numbers and emails we’ve used up to this point and throw them in a .txt file that we can search with. Copy the text below and paste it into a terminal window.

cat << EOF >> regex-test.txt
1234567891
123 456 7891
123 456-7891
123-456-7891
(123) 456 7891
(123) 456-7891
regular-expression-master101@gmail.com
joe_smith@aol.com
business-man99@bigbusiness.net
EOF

This will create a new file called regex-test.txt and the phone numbers and emails will be inside. From here, we can use grep to search for matches.

For my version of grep (2.5.1), the \s doesn’t always work to identify whitespace. Because of this, you may or may not need to replace the instances of \s with [:space:] or just an actual space. With this in mind, here is an updated regular expression to use with grep to search phone numbers. Notice that I have changed \s to [:space:] and removed the ^ and $.

Phone number regex: for use with grep
\(?[0-9]{3}\)?[-[:space:]]?[0-9]{3}[-[:space:]]?[0-9]{4}

Putting it all together, run this command in the same directory as the regex-test.txt file you just created. If it doesn’t work, try replacing the space with either \s or just actual whitespace.

grep -E '\(?[0-9]{3}\)?[-[:space:]]?[0-9]{3}[-[:space:]]?[0-9]{4}' regex-test.txt
